# Evaluation: 0015 — Lease-Renewal Heartbeat as the Liveness Signal

Verdict: FAIL
Round: 1
Reviewed against: ADR 0014 (Accepted/immutable), spec 04 §"Multi-session coordination" (frozen, ADR 0005), ADR 0001/0003, ADR/README.md, and `.docs/evaluations/multi-session-lock-helper-review-findings.md` (round-3 re-review).

## Findings

- [BLOCKER] **The thin orchestrator cannot renew while blocked in a long sub-agent call, so a legitimately-busy session's own lease goes stale mid-slice and a peer reclaims (and `git worktree remove -f`s) its in-progress work — the exact double-grant the helper exists to prevent, now self-inflicted.** §2 makes renewal "a driver-loop responsibility of the session (the thin orchestrator)" on a ~TTL/3 cadence. But a `/loom:run` session is single-threaded: while it is suspended inside one long developer/eval sub-agent Task call (20–40+ min — ADR 0012/0013, and the norm for this workload) it executes nothing and *cannot* emit a heartbeat or acquire the §2 lock to refresh the lease. The renew cadence can only fire *between* driver-loop iterations, i.e. at sub-agent-call boundaries; during a single 40-min call, **zero** renewals occur.

  Concrete interleaving: Session A claims slice X, lease-timestamp `T0`; slice-lease TTL = 15 min (a value fully permitted by this ADR, which fixes only cadence ≈ TTL/3 and defers the numeric TTL). A dispatches the developer pass — one Task call that blocks 40 min. At `T0+15` A's lease is stale, though A is very much alive (its sub-agent *is* the progress). Peer B scans, sees X stale, takes the §2 lock, evaluates liveness = lease-freshness = **STALE → "holder is dead,"** runs `git worktree remove -f` on A's worktree, clears the Active entry, and claims X. At `T0+40` A's sub-agent returns to a removed worktree and a slice now owned/being-implemented by B: double-grant **plus** destroyed in-progress work — strictly worse than the round-2 pid-reap it claims to fix.

  The ADR does not foreclose this: it neither (a) requires the slice-lease TTL to **exceed the longest expected blocking sub-agent op**, nor (b) provides an **out-of-band/background renewer**, nor (c) ties liveness to a signal that **survives a blocked session**. Its stated framing — "~TTL/3 … comfortably below the TTL with margin for clock skew and a *single missed beat*" (§2, Notes) — actively points *away* from the only in-model fix (TTL > longest op): a 40-min block at a TTL/3 cadence is many missed beats, not one. Note this reintroduces the round-2 "reap the living" failure in a new guise: ADR 0014's superseded **membership** signal survived a blocked session (the `wt-<sid>` dir persists through the block), which is exactly the property that protected a legitimately-busy holder; §3's replacement removes that property without substitute. The Consequences claim that lease-freshness "closes the three-round oscillation" and "a live session proves liveness by continuing to renew" therefore does not hold for a thin orchestrator — a live session provably *cannot* renew during its longest and most valuable operations.

- **Supersession scope is precise and does not weaken any non-liveness invariant.** §5 enumerates the preserved ADR 0014 decisions (session-owned worktrees; the §2 lock with `mkdir`/ref-CAS + rename-capture + lock-TTL + ADR 0008 backoff; the claim/lease registry + check-then-act; per-session `.git/` state; cold-restart re-adoption; the three locked shared-`main` writes) accurately; the supersession is limited to the liveness signal, mirroring the 0011→0010 / 0007→0001 precedent.
- **ADR 0014's body is untouched** and **spec 04 is not edited** (verified: the 0015 commit `a67933c` touches only `0015-…md` and `README.md`; the spec-04 amendment is explicitly deferred as a later frozen-spec planning pass per ADR 0005).
- **Membership and the ephemeral pid are correctly demoted** (§4), consistent with the T2/T6 evidence (membership classifies a crashed-with-lingering-worktree session alive-forever; the per-invocation CLI pid is always dead by probe time).
- **Reclaim cleanup closes T2/T6**: §4 drives orphan-worktree removal off lease-staleness under the lock, making the T6 path reachable and closing the T2 wedge — *given* a correct liveness signal (which the BLOCKER puts in question).
- **README supersede-listing convention** is followed (0014 line annotated "liveness signal superseded by 0015"; 0015 in "In Review"). Traceable to the review-findings evidence; ADR 0001/0003 preservation (session-id out-of-band, never author metadata) is stated correctly.

## Required changes (for FAIL)

1. Resolve the blocked-orchestrator renewal gap at the **principle** level (numeric TTL may stay deferred). Pick and state one of: (a) **the slice-lease TTL MUST exceed the longest expected blocking sub-agent op** so the lease stays fresh across a single suspended call, and re-frame the cadence rationale accordingly (not "single missed beat / clock skew"); **and** acknowledge the resulting trade-off — a genuinely-crashed session's slice is unreclaimable for up to that TTL; or (b) mandate an **out-of-band/background renewer** that beats independently of the driver loop's suspension (name the mechanism — loom currently has only event-fired hooks, no timer/daemon); or (c) retain a **block-surviving liveness fallback** (e.g. keep membership or a stable long-lived session pid as a secondary "alive" signal so a blocked-but-present holder is not reclaimed), and reconcile it with the T2/T6 closure. In all cases, correct the Consequences "closes the three-round oscillation" claim to hold for a session that is silent-by-architecture during its long ops.
2. Fix the `§3/§F5` cross-reference (MINOR).
3. Add T4 to the mechanical carry-forward list, or reword the list to defer to the review-findings artifact (MINOR).

## Notes

The decision's direction (liveness = positive proof of ongoing progress) is well-motivated against the evidence, and the "neither pid-aggressive nor membership-lenient" framing is the right diagnosis of the oscillation. The gap is that "ongoing progress" for a thin orchestrator is carried by a *sub-agent* the orchestrator is blocked on, not by the orchestrator's own executable heartbeat — so the heartbeat must either outlast the block (TTL sizing), be emitted by something other than the suspended driver loop, or be backstopped by a block-surviving signal. Until one of those is chosen, lease-freshness is pid-aggressive-equivalent during precisely the windows that matter most.
